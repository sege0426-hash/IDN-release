Operational Documentation

IDN 2.0 최고 관리자 사용자 매뉴얼

1. 개요

본 문서는 ㈜클라우드플러그에서 개발한 IDN 서비스 사용자를 대상으로 시스템의 주요 기능과 화면 구성, 그리고 각 기능별 사용 방법을 이해하기 쉽도록 설명하기 위해 작성된 문서이다.

최고 관리자는 전체 조직 범위 또는 특정 조직 범위를 선택하여 데이터를 조회할 수 있으며, 조직 관리, 라이선스 관리, 보안 정책, 시스템품질 메뉴에 접근할 수 있다.

실제 운영에서는 대시보드 점검 후 조직/사용자 구조 정비, 네트워크 자원 관리, 보안 정책 점검, 시스템 품질 검증 순으로 진행하는 것이 안전하다.

1.1 작성 목적

본 문서의 목적은 IDN 서비스에서 제공하는 주요 기능을 사용자가 정확하게 이해하고, 실제 업무 환경에서 원활하게 활용할 수 있도록 지원하는 데 있다.

2. 로그인

최고 관리자는 로그인 화면에서 IDN 계정 아이디와 비밀번호를 입력한 후 로그인한다.
인증 결과에 따라 대시보드로 바로 이동하거나 2단계 인증을 진행한다.
최초 로그인 혹은 비밀번호 재설정이 필요한 경우에는 비밀번호 재설정 화면으로 이동한다.

2.1 로그인

▶메뉴 위치: 로그인 화면

▶기능 설명:

관리자 계정을 이용하여 시스템에 접속한다.

입력한 계정 정보에 대한 인증을 수행하며, 인증 결과에 따라 관리자 대시보드 화면 또는 추가 인증 화면으로 이동한다.

▶사용 절차:

1. 로그인 화면에 접속한다.

2. 아이디와 비밀번호를 입력한다.

3. 로봇 확인(자동화 방지) 항목을 선택한다.

4. 로그인 버튼을 클릭한다.

5. 인증 결과에 따라 다음 화면으로 이동한다.

정상 인증 시: 대시보드 화면

추가 인증 시: 2단계 인증 화면

비밀번호 변경 필요 시: 비밀번호 재설정 화면

▶입력 항목:

항목 설명 입력 가이드
아이디 관리자 계정 ID 필수 입력: 대소문자 구분 필요
비밀번호 계정 비밀번호 필수 입력: 대소문자 구분 필요

▶처리 결과 / 유의사항:

로그인 실패 시 오류 메시지가 표시된다.

비밀번호 변경이 필요한 계정은 로그인 후 비밀번호 재설정 화면으로 이동한다.

로그인 후 내부 데이터 동기화 과정이 수행되며 이로 인해 대시보드 전환이 지연될 수 있다.

접근 허용 장치 IP가 등록되지 않았거나 비밀번호가 만료된 경우 로그인이 제한될 수 있다.

로그인 실패 시 화면에 표시되는 메시지를 확인한 후, 필요한 경우 관리자에게 문의하여 계정 상태를 확인해야 한다.

▶스크린샷:

[로그인 화면]

2.2 2단계 인증

▶메뉴 위치: 로그인 후 2단계 인증 화면

▶기능 설명:

아이디/비밀번호 인증 이후 추가 인증이 필요한 계정에 대해 인증코드를 입력하여 사용자 본인 여부를 확인한다.

▶사용 절차:

1. 로그인 후 2단계 인증 화면으로 이동한다.

2. 인증코드 요청 버튼을 클릭하여 인증코드를 발급받는다.

3. 수신한 인증코드를 입력한다.

4. 인증 완료 시 다음 화면으로 이동한다.

▶화면 내 주요 버튼:

버튼명 설명
인증코드 요청 인증코드를 신규 발급하거나 재발급한다.

▶처리 결과 / 유의사항:

인증 코드의 유효시간은 발급 시점 기준 3분이며 유효시간이 초과된 경우 재발급이 필요하다.

인증코드는 1회만 사용할 수 있으며, 재발급 시 기존 코드는 무효화된다

인증 실패 시 메시지가 화면에 표시되며 코드를 다시 확인하며 입력해야 한다.

인증코드를 수신하지 못한 경우, 재발급을 시도하거나 사전에 등록된 연락 수단(이메일, SMS 등)을 확인해야 한다.

지속적으로 인증에 실패하거나 접근이 제한되는 경우, 관리자에게 문의하여 계정 상태를 확인한다.

로그인 실패 시 화면에 표시되는 메시지를 확인한 후, 필요한 경우 관리자에게 문의하여 계정 상태를 확인해야 한다.

2.3 비밀번호 재설정

▶메뉴 위치: 로그인 후 비밀번호 재설정 화면

▶기능 설명:

최초 로그인 또는 비밀번호 변경이 필요한 계정에 대해 새로운 비밀번호를 설정한다.

비밀번호 변경이 완료되면 보안 정책에 따라 재로그인이 요구될 수 있다.

▶사용 절차:

1. 비밀번호 재설정 화면으로 이동한다.

2. 새 비밀번호와 새 비밀번호 확인을 입력한다.

3. 확인 버튼을 클릭한다.

4. 비밀번호 변경 결과를 확인한다.

▶입력 항목:

항목 설명 입력 가이드
새 비밀번호 변경할 비밀번호 비밀번호 정책에 맞게 입력
새 비밀번호 확인 변경 비밀번호 확인 새 비밀번호와 동일하게 입력

▶화면 내 주요 버튼:

버튼명 설명
확인 입력한 비밀한 비밀번호로 변경을 수행한다.

▶처리 결과 / 유의사항:

입력한 비밀번호는 시스템의 보안 정책(길이, 문자 조합 등)을 검증한 후 저장된다.

새 비밀번호와 확인 값이 일치하지 않을 경우 변경이 진행되지 않으며 오류 메시지가 표시된다.

보안을 위해 일정 횟수 이상 실패 시 변경이 제한될 수 있다.

비밀번호 변경이 완료되면 보안 정책에 따라 재로그인이 요구될 수 있다.

변경된 비밀번호는 이후 로그인 시 사용해야 한다.

▶스크린샷:

[비밀번호 재설정 화면]

5. 기능별 상세 설명

5.1 대시보드

▶메뉴 위치: 좌측 메뉴 > 대시보드

▶기능 설명:

선택한 조직(그룹) 범위를 기준으로 시스템 운영 현황을 요약하여 제공한다.

알람 현황, 사용자 현황, VNET 기준 통계, 최근 감사 로그 등을 한 화면에서 확인할 수 있으며, 주요 지표를 통해 현재 상태를 빠르게 파악할 수 있다.

대시보드 내 차트 및 항목을 클릭하면 관련 관리 화면으로 이동하여 상세 내용을 확인할 수 있다

▶사용 절차:

1. 좌측 메뉴에서 대시보드를 선택한다.

2. 필요 시 상단 그룹 컨텍스트를 변경하여 조회 범위를 설정한다.

3. 카드, 차트, 최근 로그 영역을 통해 주요 현황을 확인한다.

4. 추가 확인이 필요한 항목을 클릭하여 관련 상세 화면으로 이동한다.

구분 설명
알람 현황 발생한 알람의 상태 및 추이를 차트 형태로 제공하며, 클릭 시 알람 관리 화면으로 이동한다.
사용자 현황 총 사용자 수, 접속 사용자 수, 차단 사용자 수, 등록 사용자 수 등을 표시한다.
VNET 통계 가상보안네트워크의 최근 접속 사용자 수를 표시한다.
최근 감사 로그 최근 수행된 작업 이력을 시간순으로 확인할 수 있다.

▶처리 결과 / 유의사항:

상단 조직 컨텍스트 변경 시, 해당 범위 기준으로 대시보드 데이터가 재조회된다.

대시보드는 실시간 알람 반영을 위해 SSE기반 구독이 동작하며, 새로운 알람이 발생할 경우 화면에 자동 반영된다.

차단 사용자 수가 급증하면 계정 보안 정책과 IP 접근 제어 정책을 함께 점검한다.

최근 감사 로그를 함께 확인하면, 특정 상태 변화에 대한 원인(선행 작업 및 변경 주체)을 빠르게 파악할 수 있다.

▶스크린샷:

[대시보드 화면]

5.2 컨트롤러 관리

컨트롤러는 가상 네트워크(VNET)를 구성하고 운영하기 위한 핵심 서버이다.
각 조직별 네트워크 환경을 구성하기 위해서는 최소 1개 이상의 컨트롤러가 반드시 등록되어 있어야 한다.

1. 컨트롤러 등록 버튼을 클릭한다.

4. 등록 완료 후 목록에 신규 컨트롤러가 정상적으로 반영되었는지 확인한다.

등록 완료 후, 컨트롤러 상태(정상/비정상)를 함께 확인하여 정상적으로 연결되었는지 점검해야 한다.

▶입력 항목:

항목 설명 입력 가이드

- 각 자릿수는 0~255 범위 내 숫자만 가능
- 아래 IP 주소 사용 불가

5.2.4 컨트롤러 삭제

▶기능 설명:

선택한 컨트롤러를 시스템에서 삭제한다.

삭제된 컨트롤러는 더 이상 VNET 구성 및 네트워크 운영에 사용할 수 없다.

▶사용 절차:

1. 컨트롤러 관리 화면에서 삭제할 컨트롤러를 선택한다.

4. 삭제 완료 후 목록에서 해당 컨트롤러가 제거되었는지 확인한다.

▶처리 결과 / 유의사항:

삭제된 컨트롤러는 복구할 수 없으므로, 삭제 전 반드시 대상이 올바른지 확인해야 한다.

운영 중인 컨트롤러를 삭제할 경우, 해당 컨트롤러를 통해 연결된 네트워크 통신이 중단될 수 있다.

삭제 시 확인 창을 통해 최종 검증 절차가 수행되며, 사용자의 명시적 확인 이후 삭제가 진행된다.

삭제 완료 후, 관련 서비스 상태 및 연결 상태를 함께 점검해야 한다.

5.2.5 컨트롤러 Linux 패키지 준비 및 설치

리눅스 환경에 배포할 컨트롤러 실행 패키지를 생성하고, 대상 호스트에 설치하여 서비스 형태로 실행한다.

설치 완료 후 컨트롤러는 IDN과 연동되어 네트워크 제어 기능을 수행한다.

1. Linux 빌드 장비에서 소스 디렉토리로 이동한다.

cd /path/to/IDNclient_wg

2. 패키지를 생성한다.

컨트롤러 단독 패키지의 경우:

./scripts/package-controller.sh

컨트롤러 + 중계서버 통합 패키지의 경우:

./scripts/package-linux-appliances.sh all

3. 생성된 패키지 파일을 확인한다.

dist/idn-controller-.tar.gz

4. 생성된 패키지를 대상 호스트로 복사한다.

5. 대상 호스트에서 작업 디렉토리를 생성하고 압축을 해제한다.

mkdir -p /tmp/idn-controller-package
tar xzf idn-controller-.tar.gz -C /tmp/idn-controller-package

6. 설치 스크립트를 실행한다.

cd /tmp/idn-controller-package
sudo ./install.sh --auth-ip --auth-port

7. 실행 예시

sudo ./install.sh --auth-ip 61.109.220.75 --auth-port 15004

▶대상 호스트 요구 사항:

항목 설명
init 시스템 `systemd` 사용 환경
쉘 `bash` 사용 환경
권한 `sudo` 사용 가능 환경
패키지 관리자 apt-get 기준 자동 의존성 설치 지원 (미사용 환경은 수동 설치 필요)

▶컨트롤러 런타임 의존성:

구분 패키지
라이브러리 `libevent`, `libssl`, `jansson`, `glib2`, `libblkid`

대상 환경에 따라 사전 설치가 필요할 수 있다.

▶설치 결과 경로:

항목 경로
실행 바이너리 `/opt/idn/controller/bin/idn-controllerd`
래퍼 명령 `/usr/local/bin/idn-controller`
서비스 파일 `/etc/systemd/system/idn-controller.service`
인증 서버 설정 `/etc/default/idn-controller.auth`
ID 환경 파일 `/etc/default/idn-controller`

▶인증 서버 설정 형식:

컨트롤러는 /etc/default/idn-controller.auth 파일을 통해 인증 서버 정보를 읽는다.

기본 형식

[netif]

예시

61.109.220.75 15004
61.109.220.75 15004 enp1s0

netif(네트워크 인터페이스)를 생략할 경우, 기본 라우트 인터페이스를 자동으로 감지한다.

설정 파일의 첫 번째 유효 행은 반드시 인증 서버 정보여야 하며, 빈 줄이나 주석은 무시된다.

파일이 비어 있거나 주석만 존재하는 경우, 컨트롤러 서비스는 시작되지 않는다.

▶처리 결과 / 유의사항:

설치 완료 후 systemctl을 통해 서비스 상태를 반드시 확인해야 한다.

systemctl status idn-controller

서비스가 정상적으로 실행되지 않는 경우 다음 항목을 점검한다.

인증 서버 IP 및 포트 설정값

네트워크 인터페이스 설정

런타임 의존성 라이브러리 설치 여부

방화벽 또는 네트워크 정책에 의해 Orchestrator와 통신이 차단될 수 있으므로, 관련 포트가 허용되어 있는지 확인해야 한다.

설치 경로 및 설정 파일을 임의로 변경할 경우 서비스 실행에 문제가 발생할 수 있으므로 권장 경로를 유지해야 한다.

5.2.6 컨트롤러 Linux 실행, 업데이트 및 점검

▶기능 설명:

설치된 컨트롤러 서비스를 실행하고, 상태 확인, 로그 조회, 업데이트 및 제거 작업을 수행한다.

운영 중에는 서비스 상태와 로그를 통해 정상 동작 여부를 지속적으로 점검해야 한다.

▶사용 절차:

1. 컨트롤러 ID를 등록하고 서비스를 시작한다.

sudo idn-controller

2. 서비스 상태 및 제어 명령을 사용하여 운영 상태를 관리한다.

상태 확인:

idn-controller status

서비스 중지:

sudo idn-controller stop

서비스 재시작:

sudo idn-controller restart

로그 확인:

idn-controller logs

컨트롤러를 업데이트하는 경우:

(패키지를 /tmp/idn-controller-package에 해제 후)
sudo ./install.sh --skip-deps
sudo idn-controller restart

컨트롤러를 제거한다.

컨트롤러를 제거하는 경우

sudo ./uninstall.sh

설정 파일까지 함께 삭제할 경우

sudo KEEP_CONFIG=0 ./uninstall.sh

▶점검 및 장애 대응:

인증 파일 설정 오류 발생 시

cat /etc/default/idn-controller.auth

첫 번째 유효 행이 <orchestrator-ip> <auth-port> 형식인지 확인한다.
(예: 61.109.220.75 15004)

서비스 시작 실패 시

systemctl status idn-controller.service
journalctl -u idn-controller.service -n 200 --no-pager

명령을 통해 서비스 상태 및 로그를 확인한다.

인증 서버 주소 변경 시

sudo vi /etc/default/idn-controller.auth
sudo idn-controller restart

설정 파일 수정 후 서비스를 재시작한다.

▶처리 결과 / 유의사항:

컨트롤러 실행 후, 오케스트레이터에 정상 등록 및 승인 상태인지 반드시 확인해야 한다.

로그 확인은 장애 원인 분석의 기본 절차이며, 문제 발생 시 우선적으로 수행해야 한다.

업데이트 시 --skip-deps 옵션은 기존 의존성이 정상 설치되어 있는 경우에만 사용해야 한다.

제거 시 기본적으로 설정 파일은 유지되며, 완전 삭제가 필요한 경우에만 KEEP_CONFIG=0 옵션을 사용한다.

운영 중 서비스 재시작은 연결 세션에 영향을 줄 수 있으므로, 작업 시간 및 영향 범위를 고려하여 수행해야 한다.

▶권장 운영 흐름:

3. install.sh 실행

4. sudo idn-controller <controller_id> 실행

5.3 중계서버 관리

중계서버는 가상 네트워크(VNET)에 연결된 단말 간 통신을 지원하는 릴레이(중계) 전용 서버이다.
직접 연결이 어려운 네트워크 환경에서 데이터 전달 경로를 제공하여 통신 안정성을 확보한다.

중계서버의 신규 등록, 정보 수정, 삭제 기능을 제공한다.

각 중계서버에 대해 운영 상태, 등록 상태, 외부/내부 IP 및 포트 정보를 확인할 수 있으며, 이를 통해 통신 경로 및 연결 상태를 점검할 수 있다.

▶사용 절차:

1. 좌측 메뉴에서 중계서버 관리 메뉴로 이동한다.

2. 검색 조건을 입력하여 중계서버 목록을 조회한다.

3. 중계서버 등록 버튼을 클릭한다.

4. 필요한 정보를 입력한 후 저장한다.

5. 목록에서 대상 중계서버를 선택하여 수정 또는 삭제를 수행한다.

6. 작업 결과가 정상적으로 반영되었는지 확인한다.

▶입력 항목:

항목 설명 입력 가이드
검색 대상 검색 기준 중계서버 이름 기준으로 선택
검색어 조회 키워드 이름의 일부만 입력해도 검색 가능

삭제 선택한 중계서버를 삭제한다.

▶처리 결과 / 유의사항:

상단 조직 컨텍스트에 따라 조회 범위가 결정되며, 선택한 그룹 기준으로 목록이 표시된다.

중계서버는 단말 간 통신 경로에 직접적으로 관여하므로, 설정 변경 시 통신 품질 및 연결 상태에 영향을 줄 수 있다.

외부 IP/포트 및 내부 IP/포트 설정이 올바르지 않은 경우, 릴레이 통신이 정상적으로 동작하지 않을 수 있다.

중계서버 삭제 시 해당 서버를 사용하는 VNET 또는 단말 연결에 영향이 발생할 수 있으므로, 사전에 연관 관계를 반드시 확인해야 한다.

등록 및 수정 후에는 반드시 서버 상태(정상/비정상)를 확인하여 정상 동작 여부를 점검해야 한다.

5.3.4 중계서버 삭제

▶기능 설명:

운영 중인 중계서버를 삭제할 경우, 해당 서버를 통해 연결된 통신이 즉시 중단될 수 있다.

삭제 완료 후, 관련 서비스 상태 및 연결 상태를 함께 점검해야 한다.

5.3.5 중계서버 Linux 패키지 준비 및 설치

리눅스 환경에 배포할 중계서버 실행 패키지를 생성하고, 대상 호스트에 설치하여 서비스 형태로 실행한다.

설치 완료 후 중계서버는 IDN과 연동되어 단말 간 통신 중계 기능을 수행한다.

1. Linux 빌드 장비에서 소스 디렉토리로 이동한다.

cd /path/to/IDNclient_wg

2. 패키지를 생성한다.

중계서버 단독 패키지의 경우:

./scripts/package-relay.sh

컨트롤러 + 중계서버 통합 패키지의 경우:

./scripts/package-linux-appliances.sh all

3. 생성된 패키지 파일을 확인한다.

dist/idn-relay-<YYYYMMDD>.tar.gz

4. 생성된 패키지를 대상 호스트로 복사한다.

5. 대상 호스트에서 작업 디렉토리를 생성하고 압축을 해제한다.

mkdir -p /tmp/idn-relay-package
tar xzf idn-relay-<YYYYMMDD>.tar.gz -C /tmp/idn-relay-package

6. 설치 스크립트를 실행한다.

cd /tmp/idn-relay-package
sudo ./install.sh --auth-ip <orchestrator-ip> --auth-port <auth-port>

7. 실행 예시

sudo ./install.sh --auth-ip 61.109.220.75 --auth-port 15004

▶대상 호스트 요구 사항:

항목 설명
init 시스템 `systemd` 사용 환경
쉘 `bash` 사용 환경
권한 `sudo` 사용 가능 환경
패키지 관리자 apt-get 기준 자동 의존성 설치 지원 (미사용 환경은 수동 설치 필요)

▶중계서버 런타임 의존성:

구분 패키지
라이브러리 `libevent`, `libgtop`, `libssl`, `jansson`, `glib2`, `libblkid`, `wireguard-tools`, `iproute2`

대상 환경에 따라 사전 설치가 필요할 수 있다.

▶설치 결과 경로:

항목 경로

중계서버는 `/etc/default/idn-edge.auth` 파일을 통해 인증 서버 정보를 읽는다.

기본 형식

<orchestrator-ip> <auth-port> [netif]

예시

61.109.220.75 15004 enp1s0

netif를 생략하면 기본 라우트 인터페이스를 자동 감지한다.

첫 번째 유효 행은 반드시 인증 서버 정보여야 한다.

파일이 비어 있거나 주석만 존재하는 경우 서비스가 시작되지 않는다.

systemctl status idn-relay

서비스가 정상 동작하지 않을 경우 인증 서버 설정, 네트워크 인터페이스, 의존성 설치 여부를 점검한다.

5.4.1 VNET 생성

▶기능 설명:

신규 가상보안네트워크(VNET)를 생성한다.

생성된 VNET은 컨트롤러 및 중계서버와 연동되어 네트워크 통신 환경을 구성한다.

▶사용 절차:

1. VNET 생성 버튼을 클릭한다.

2. 네트워크 정보를 입력한다.

3. 저장 버튼을 클릭하여 생성을 완료한다.

4. 생성 완료 후 목록에 정상적으로 반영되었는지 확인한다.

▶입력 항목:

항목 설명 입력 가이드
VNET 이름 네트워크 식별 고유 이름 - 영문, 숫자, 일부 특수문자(-)만 사용 가능
- 중복 불가 및 생성 후 수정 불가
VNET 별칭 네트워크 별칭 선택 입력
컨트롤러 네트워크 관리 서버 선택 사전에 등록된 컨트롤러 중 1개 선택
중계서버 통신 보조 서버 사용 여부 필요 시 등록된 중계서버 선택
암호화 설정 데이터 암호화 규격 선택 - 알고리즘: LEA256, ARIA256, AES256 중 선택
- 모드: CBC, GCM 중 선택
네트워크 주소(CIDR) 가상 IP 대역 설정 - 자동: 시스템에서 중복되지 않는 주소 임의 배정
- 수동: 8, 16, 24 비트 단위 중 선택하여 직접 입력
삭제 예정일 네트워크 자동 삭제 예약 필요 시 날짜 및 시간 지정
단말 접속 장치 연결 허용 여부 활성 시 단말의 네트워크 참여 허용
외부 트래픽 외부 인터넷 통신 허용 여부 활성 시 외부 네트워크 통신 허용
기기 인증 접속 기기 추가 인증 적용 여부 활성 시 단말 인증 절차 적용

▶처리 결과 / 유의사항:

VNET 이름은 시스템 내 고유 식별 값으로, 생성 이후 변경할 수 없다.

컨트롤러는 반드시 선택해야 하며, 선택된 컨트롤러를 기준으로 네트워크가 구성된다.

중계서버를 설정하지 않은 경우, 네트워크 환경에 따라 단말 간 직접 통신이 제한될 수 있다.

CIDR 설정 시 기존 네트워크와 주소가 중복되지 않도록 주의해야 하며, 잘못 설정할 경우 통신 충돌이 발생할 수 있다.

암호화 설정은 네트워크 보안 수준과 성능에 영향을 미치므로 운영 환경에 맞게 선택해야 한다.

단말 접속, 외부 트래픽, 기기 인증 설정은 네트워크 접근 정책에 직접적인 영향을 미치므로 신중하게 설정해야 한다.

삭제 예정일이 설정된 경우, 지정된 시점에 자동으로 네트워크가 삭제되므로 운영 중인 네트워크인지 반드시 확인해야 한다.

생성 후에는 반드시 네트워크 상태 및 연결 여부를 점검해야 한다.

5.4.3 VNET 삭제

▶기능 설명:

선택한 VNET을 시스템에서 삭제한다.

삭제된 VNET은 더 이상 네트워크 통신에 사용할 수 없으며, 관련된 모든 연결이 해제된다.

▶사용 절차:

1. VNET 목록 화면에서 삭제할 VNET을 선택한다.

2. 삭제 버튼을 클릭한다.

3. 확인 창에서 삭제 여부를 확인한 후 삭제를 확정한다.

4. 삭제 완료 후 목록에서 해당 VNET이 제거되었는지 확인한다.

▶처리 결과 / 유의사항:

삭제된 VNET은 복구할 수 없으므로, 삭제 전 대상이 정확한지 반드시 확인해야 한다.

해당 VNET에 연결된 엔드포인트 및 관련 리소스는 모두 연결이 해제되며, 통신이 중단된다.

항목 설명 입력 가이드

5. 할당 결과가 정상적으로 반영되었는지 확인한다.

사용자 할당이 완료되면 해당 엔드포인트는 지정된 사용자와 연결되어 사용된다.

할당 이후에는 사용자 및 엔드포인트 상태를 함께 확인하여 정상적으로 반영되었는지 점검해야 한다.

3. 그래프 영역에서 확인할 대상을 선택한다.

4. 상세 패널에서 상태 및 메모 정보를 확인한다.

5. 필요 시 자원을 드래그하여 위치를 조정하거나, 자동 배치 기능을 실행한다.

2. 자유 배치: 지도 상의 엔드포인트 아이콘을 마우스로 드래그하여 원하는 위치로 이동할 수 있다.이를 통해 관리하기 편한 형태로 네트워크 구조를 시각적으로 정리할 수 있다.

3. 자동 배치: 네트워크 구성이 복잡한 경우 자동 배치 버튼을 클릭하여 전체 그래프를 정렬할 수 있다.

▶처리 결과 / 유의사항:

드래그 생성 시 선택한 역할(Gateway/Host)은 생성 이후 변경이 제한될 수 있으므로 신중하게 선택해야 한다.

자유 배치는 화면 상의 위치만 변경되며, 실제 네트워크 연결 구조나 통신에는 영향을 주지 않는다.

자동 배치 실행 시 기존 수동 배치 위치는 초기화되므로 필요 시 다시 조정해야 한다.

조치 내용은 이후 운영 및 감사에 활용되므로, 구체적으로 작성하는 것이 권장된다.

코멘트 내용은 운영 및 감사 이력으로 활용되므로, 명확하고 구체적으로 작성하는 것이 권장된다.

5.10 이상감지 관리

▶메뉴 위치: 좌측 메뉴 > 관리 > 이상감지 관리

▶기능 설명:

시스템 운영 중 발생할 수 있는 보안 위협, 장애, 리소스 과부하 상황을 감지하기 위한 임계치를 설정한다.

설정된 기준에 따라 이상 징후 발생 시 자동으로 알람을 생성하여 운영자가 즉시 대응할 수 있도록 지원한다.

▶알람 연동 및 확인:

이상감지 관리에서 설정한 주의, 경고, 긴급 단계의 기준값에 도달하면 시스템은 자동으로 알람을 발생시킨다.

발생한 알람은 알람 관리 메뉴에서 실시간으로 확인할 수 있다.

▶이상감지 분류 및 설정 항목:

분류 주요 설정 및 감지 항목 알람 발생 목적
보안 비정상 접근 시도 수, 외부 트래픽(LAN→WAN) 감시, 이상 트래픽 감지 비인가 접근 및 정보 유출 시도 탐지
장애 오류 발생률, 서비스 응답 지연 시간 서비스 품질 저하 및 장애 조기 감지
리소스 CPU 사용률, 메모리 사용률 자원 부족으로 인한 시스템 장애 예방

▶공통 설정 방법:

각 항목은 위험 수준에 따라 3단계로 수치를 입력하며, 설정된 기준을 초과하면 해당 수준에 맞는 심각도의 알람이 생성된다.

항목 설명
주의 초기 이상 징후 발생 단계 (관찰 필요)
경고 즉각적인 원인 분석 및 모니터링 필요
긴급 시스템 장애 또는 보안 침해로 즉시 조치 필요

▶처리 결과 / 유의사항:

조직 컨텍스트에 따라 설정 범위가 적용되며, 선택한 조직 기준으로 감지 정책이 저장된다.

설정된 임계값을 초과하는 이벤트가 발생하면, 알람 관리 메뉴에 자동으로 기록된다.

임계값이 너무 낮으면 오탐(false positive)이 증가하고, 너무 높으면 실제 이상 징후를 놓칠 수 있다.

이상감지 설정 변경 후에는 알람 발생 패턴을 지속적으로 모니터링하여 기준값을 보정하는 것이 권장된다.

설정 변경 후에는 실제 알람 발생 여부를 모니터링하여 기준값을 지속적으로 조정하는 것이 권장된다.

5.10.2 장애 이상감지 설정

▶기능 설명:

▶사용 절차:

1. 좌측 메뉴에서 관리 > 이상감지 관리 > 장애로 이동한다.

2. 각 감지 항목의 임계값을 입력하거나 슬라이더(드래그)를 통해 설정한다.

3. 저장 버튼을 클릭하여 설정을 반영한다.

4. 설정 이후 알람 발생 여부를 확인하여 기준값의 적절성을 점검한다.

▶주요 설정 항목:

서비스 오류율: 전체 요청 수 대비 발생하는 에러 비율(%)을 설정한다.

응답 지연 시간: 시스템 응답이 평소보다 늦어질 경우의 대기 시간(ms) 기준을 설정한다.

연결 상태: 컨트롤러 및 주요 장비와의 통신 단절 감지 기준을 설정한다.

▶처리 결과 / 유의사항:

설정한 임계값을 초과할 경우, 설정된 단계(주의/경고/긴급)에 따라 알람이 자동 생성된다.

조직 컨텍스트에 따라 설정이 적용되므로, 작업 전 대상 조직을 반드시 확인해야 한다.

서비스 오류율 기준이 낮을 경우 일시적인 오류에도 빈번한 알람이 발생할 수 있다.

응답 지연 시간 기준은 시스템 평균 응답 시간 대비 적절한 수준으로 설정해야 한다.

연결 상태 감지는 네트워크 환경에 영향을 받을 수 있으므로, 일시적인 단절과 실제 장애를 구분할 수 있도록 기준을 설정해야 한다.

설정 변경 후에는 알람 발생 패턴을 지속적으로 모니터링하여 기준값을 보정하는 것이 권장된다.

5.10.3 리소스 이상감지 설정

▶기능 설명:

시스템 자원 사용량에 대한 임계값을 설정하여 과부하 및 자원 부족 상태를 사전에 감지한다.

1. 좌측 메뉴에서 관리 > 이상감지 관리 > 리소스로 이동한다.

3. 저장 버튼을 클릭하여 설정을 반영한다.

4. 설정 이후 알람 발생 여부를 확인하여 기준값의 적절성을 점검한다.

▶주요 설정 항목:

디스크/IP 자원: 저장 공간 및 가용 가상 IP 주소의 잔여량 부족 상태를 감지한다.

▶처리 결과 / 유의사항:

설정한 임계값을 초과할 경우, 설정된 단계(주의/경고/긴급)에 따라 알람이 자동 생성된다.

조직 컨텍스트에 따라 설정이 적용되므로, 작업 전 대상 조직을 반드시 확인해야 한다.

CPU 및 메모리 기준이 너무 낮을 경우 정상적인 부하 상황에서도 알람이 빈번하게 발생할 수 있다.

디스크 및 IP 자원은 부족 상태가 발생하면 서비스 중단으로 이어질 수 있으므로, 여유 용량을 고려하여 기준을 설정해야 한다.

리소스 사용량은 시간대 및 운영 패턴에 따라 변동되므로, 일정 기간 모니터링 후 기준값을 조정하는 것이 권장된다.

5.12 요청 관리

▶메뉴 위치: 좌측 메뉴 > 관리 > 요청 관리

▶기능 설명:

일반 사용자가 신청한 VNET 및 엔드포인트 관련 변경 요청을 조회하고, 승인 또는 거절을 수행한다.

요청 대상에는 생성, 수정, 삭제, 사용자 할당/해제 등 모든 자원 변경 행위가 포함되며, 승인 완료 후에만 실제 시스템에 반영된다.

▶결재 프로세스 및 연동:

시스템 보안 정책에 따라 최고 관리자 및 조직 관리자를 제외한 사용자의 모든 자원 변경 행위는 결재 승인 후 반영된다.

주요 대상 행위

VNET: 생성, 수정, 삭제

엔드포인트: 생성, 수정, 삭제, 사용자 할당 및 해제

알람 연동

요청이 생성되면 상급자에게 즉시 알람이 전송된다.

알람은 알람 관리 또는 요청 관리 메뉴에서 확인할 수 있다.

▶요청 관리 목적:

무분별한 가상 네트워크 생성이나 인가되지 않은 단말 등록을 사전에 차단한다.

자원 사용의 적정성을 검토하여 불필요한 자원 낭비를 방지한다.

요청자, 승인자, 처리 이력 등을 기록하여 운영의 투명성과 추적성을 확보한다.

3. 각 요청 항목의 상세 정보를 확인한다.

▶주요 항목:

항목 설명 표시 정보
요청 타입 요청된 작업 유형 CREATE, UPDATE, DELETE, ASSIGN 등
리소스 타입 결재 대상 자원의 종류 VNET 또는 ENDPOINT 구분 표시
타겟 요청이 적용될 대상 변경 대상 VNET 이름 또는 단말 식별 정보 표시

결과 최종 처리 상태 대기, 승인, 거절 등 현재 안건의 상태 표시

요청은 승인 완료 이후에만 실제 시스템에 반영된다.

결재 단계 정보를 통해 현재 승인 진행 상황을 확인할 수 있다.

대기 상태 요청은 우선적으로 검토하여 지연되지 않도록 관리해야 한다.

요청 거절 시 사유를 명확히 기록하여 사용자와의 혼선을 방지해야 한다.

요청 처리 결과는 운영 및 감사 이력으로 활용되므로, 정확하게 관리해야 한다.

최고 관리자의 승인은 최종 승인으로 처리되며, 추가 결재 단계 없이 즉시 반영된다.

승인 전 요청 내용(대상 자원, 변경 내용)을 반드시 확인해야 한다.

승인된 작업은 즉시 반영되므로, 운영 중인 자원에 영향을 줄 수 있다.

승인 이력은 감사 및 추적을 위해 기록되며, 이후 변경 이력 확인에 활용된다.

5. 처리 결과가 정상적으로 반영되었는지 확인한다.

최고 관리자의 반려는 최종 결정으로 처리되며, 추가 결재 단계 없이 즉시 반영된다.

반려 이력은 감사 및 추적을 위해 기록되며, 이후 운영 이력 확인에 활용된다.

5.12.4 결재 시스템 안내

▶기능 설명:

시스템의 보안과 자원 운영 통제를 위해, 최고 관리자 및 조직 관리자를 제외한 일반 사용자의 자원 변경 행위는 반드시 결재권자의 승인을 거쳐야 한다.

승인 완료된 요청만 실제 시스템에 반영되며, 모든 결재 과정은 이력으로 기록된다.

▶대상 행위:

VNET: 생성, 수정, 삭제

엔드포인트: 생성, 수정, 삭제, 사용자 할당 및 해제

▶알람 연동:

사용자가 요청을 완료하면 결재권자에게 즉시 알람이 전송된다.

알람은 알람 관리 또는 요청 관리 메뉴에서 확인할 수 있다.

▶결재권자 권한 설정:

구분 설정 옵션 역할 및 권한
중간 결재자 결재권자 체크 하위 사용자의 요청을 검토하여 승인 또는 거절 가능
최종 승인자 부서 관리자 체크 부서 내 최종 승인 권한을 가지며, 승인 시 요청이 최종 확정됨

▶결재 요청 경로:

일반 사용자의 요청은 조직도 및 직위 체계를 기반으로 상향식으로 전달된다.

기본 원칙

동일 부서 내에서 본인보다 직위가 높고 결재권자 권한이 있는 상급자에게 우선 전달된다.

단계별 승인

1차 승인 후, 요청은 다음 상위 결재권자에게 순차적으로 전달된다.

단계별 승인

1차 승인 후, 요청은 다음 상위 결재권자에게 순차적으로 전달된다.

최종 승인 단계

부서 관리자 승인 시 요청이 최종 확정된다.

부서 간 확장

부서 관리자 부재 시 상위 부서의 결재권자에게 자동 이관된다.

▶관리자 그룹 권한:

최고 관리자: 모든 조직의 결재 건에 대해 즉시 최종 승인 또는 거절 가능

조직 관리자: 소속 조직 내 모든 결재 건을 결재 경로와 관계없이 즉시 최종 승인 가능

▶처리 결과 / 유의사항:

승인되지 않은 요청은 시스템에 반영되지 않는다.

결재 경로는 조직 및 직위 구조에 따라 자동으로 결정되므로, 부서 및 직위 설정이 정확해야 한다.

결재 지연 시 요청 반영이 늦어질 수 있으므로, 결재 상태를 지속적으로 확인해야 한다.

모든 결재 이력(요청자, 승인자, 처리 시간)은 기록되며, 감사 및 운영 추적에 활용된다.

5.12.5 결재 프로세스 예시

▶기능 설명:

일반 사용자가 VNET을 생성하거나 자원 변경을 요청하면 즉시 반영되지 않고, 임시 상태로 저장된 후 결재 경로를 따라 승인 절차를 진행한다.

결재가 완료되기 전까지 해당 자원은 비활성 상태로 유지되며, 승인 이후에만 실제 시스템에 적용된다.

▶결재 흐름 개요:

요청 생성 시 자원은 임시 상태로 저장된다.

결재권자가 없는 경우 부서 관리자에게 자동 이관된다.

결재 진행 중인 자원은 수정이 제한된다.

▶실무 결재 경로 시나리오:

요청자: `test03`

1차 검토자: `test02` (동일 부서 상급자+결재권자)
